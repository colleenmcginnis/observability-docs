<ApmSupportOverview binary={true} fleet={false}>
  These configuration options are only relevant to APM Server binary users. Fleet-managed users should see the [Fleet output settings](((fleet-guide))/fleet-settings.html).
</ApmSupportOverview>

You can specify SSL/TLS options with any output that supports SSL, like ((es)), ((ls)), or Kafka.
Example output config with SSL/TLS enabled:

```yaml
output.elasticsearch.hosts: ["https://192.168.1.42:9200"]
output.elasticsearch.ssl.certificate_authorities: ["/etc/pki/root/ca.pem"]
output.elasticsearch.ssl.certificate: "/etc/pki/client/cert.pem"
output.elasticsearch.ssl.key: "/etc/pki/client/cert.key"
```

There are a number of SSL/TLS configuration options available to you:

* <DocLink slug="/en/observability/apm-configuration-ssl" section="common-configuration-options">Common configuration options</DocLink>
* <DocLink slug="/en/observability/apm-configuration-ssl" section="client-configuration-options">Client configuration options</DocLink>
* <DocLink slug="/en/observability/apm-configuration-ssl" section="server-configuration-options">Server configuration options</DocLink>

<div id="apm-ssl-common-config"></div>

## Common configuration options

Common SSL configuration options can be used in both client and server configurations.
You can specify the following options in the `ssl` section of each subsystem that
supports SSL.

<div id="apm-enabled"></div>

### `enabled`

To disable SSL configuration, set the value to `false`. The default value is `true`.

<DocCallOut title="Note">

SSL settings are disabled if either `enabled` is set to `false` or the
`ssl` section is missing.

</DocCallOut>

<div id="apm-supported-protocols"></div>

### `supported_protocols`

List of allowed SSL/TLS versions. If SSL/TLS server decides for protocol versions
not configured, the connection will be dropped during or after the handshake. The
setting is a list of allowed protocol versions:
`SSLv3`, `TLSv1` for TLS version 1.0, `TLSv1.0`, `TLSv1.1`, `TLSv1.2`, and
`TLSv1.3`.

The default value is `[TLSv1.1, TLSv1.2, TLSv1.3]`.

<div id="apm-cipher-suites"></div>

### `cipher_suites`

The list of cipher suites to use. The first entry has the highest priority.
If this option is omitted, the Go crypto library's [default suites](https://golang.org/pkg/crypto/tls/)
are used (recommended). Note that TLS 1.3 cipher suites are not
individually configurable in Go, so they are not included in this list.

<CipherSuites />

<div id="apm-curve-types"></div>

### `curve_types`

The list of curve types for ECDHE (Elliptic Curve Diffie-Hellman ephemeral key exchange).

The following elliptic curve types are available:

* P-256
* P-384
* P-521
* X25519

<div id="apm-ca-sha256"></div>

### `ca_sha256`

This configures a certificate pin that you can use to ensure that a specific certificate is part of the verified chain.

The pin is a base64 encoded string of the SHA-256 of the certificate.

<DocCallOut title="Note">
This check is not a replacement for the normal SSL validation, but it adds additional validation.
If this option is used with  `verification_mode` set to `none`, the check will always fail because
it will not receive any verified chains.
</DocCallOut>

<div id="apm-client-key-passphrase"></div>

### `key_passphrase`

The passphrase used to decrypt an encrypted key stored in the configured `key` file.

<div id="apm-client-verification-mode"></div>

### `verification_mode`

Controls the verification of server certificates. Valid values are:

<DocDefList>
  <DocDefTerm>`full`</DocDefTerm>
  <DocDefDescription>
    Verifies that the provided certificate is signed by a trusted
    authority (CA) and also verifies that the server's hostname (or IP address)
    matches the names identified within the certificate.
  </DocDefDescription>

  <DocDefTerm>`strict`</DocDefTerm>
  <DocDefDescription>
    Verifies that the provided certificate is signed by a trusted
    authority (CA) and also verifies that the server's hostname (or IP address)
    matches the names identified within the certificate. If the Subject Alternative
    Name is empty, it returns an error.
  </DocDefDescription>

  <DocDefTerm>`certificate`</DocDefTerm>
  <DocDefDescription>
    Verifies that the provided certificate is signed by a
    trusted authority (CA), but does not perform any hostname verification.
  </DocDefDescription>

  <DocDefTerm>`none`</DocDefTerm>
  <DocDefDescription>
    Performs _no verification_ of the server's certificate. This
    mode disables many of the security benefits of SSL/TLS and should only be used
    after cautious consideration. It is primarily intended as a temporary
    diagnostic mechanism when attempting to resolve TLS errors; its use in
    production environments is strongly discouraged.

    The default value is `full`.
  </DocDefDescription>
</DocDefList>

<div id="apm-server-key-passphrase"></div>

### `key_passphrase`

The passphrase is used to decrypt an encrypted key stored in the configured `key` file.

<div id="apm-server-verification-mode"></div>

### `verification_mode`

Controls the verification of client certificates. Valid values are:

<DocDefList>
  <DocDefTerm>`full`</DocDefTerm>
  <DocDefDescription>
    Verifies that the provided certificate is signed by a trusted
    authority (CA) and also verifies that the server's hostname (or IP address)
    matches the names identified within the certificate.
  </DocDefDescription>

  <DocDefTerm>`strict`</DocDefTerm>
  <DocDefDescription>
    Verifies that the provided certificate is signed by a trusted
    authority (CA) and also verifies that the server's hostname (or IP address)
    matches the names identified within the certificate. If the Subject Alternative
    Name is empty, it returns an error.
  </DocDefDescription>

  <DocDefTerm>`certificate`</DocDefTerm>
  <DocDefDescription>
    Verifies that the provided certificate is signed by a
    trusted authority (CA), but does not perform any hostname verification.
  </DocDefDescription>

  <DocDefTerm>`none`</DocDefTerm>
  <DocDefDescription>
    Performs _no verification_ of the server's certificate. This
    mode disables many of the security benefits of SSL/TLS and should only be used
    after cautious consideration. It is primarily intended as a temporary
    diagnostic mechanism when attempting to resolve TLS errors; its use in
    production environments is strongly discouraged.

    The default value is `full`.
  </DocDefDescription>
</DocDefList>

<div id="apm-server-renegotiation"></div>

### `renegotiation`

This configures what types of TLS renegotiation are supported. The valid options
are:

<DocDefList>
  <DocDefTerm>`never`</DocDefTerm>
  <DocDefDescription>
    Disables renegotiation.
  </DocDefDescription>

  <DocDefTerm>`once`</DocDefTerm>
  <DocDefDescription>
    Allows a remote server to request renegotiation once per connection.
  </DocDefDescription>

  <DocDefTerm>`freely`</DocDefTerm>
  <DocDefDescription>
    Allows a remote server to request renegotiation repeatedly.

    The default value is `never`.
  </DocDefDescription>
</DocDefList>
